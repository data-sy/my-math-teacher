---

$ARGUMENTS (마일스톤 문서 경로)를 읽고, 다음 순서로만 작업해줘. 절대 순서를 건너뛰지 말 것:

## 1단계: 문서 파싱

문서에서 다음 요소를 추출하여 리스트업:
- 언급된 모든 파일 경로 (예: ConceptRepository, application.yml)
- 패키지 경로 (예: com.mmt.api.repository)
- 클래스·어노테이션·메서드명
- 피처 플래그 key (예: mmt.migration.use-mysql-cte-for-graph)
- 성능 수치 (예: ~5ms, ~50ms)
- 의존성·라이브러리명
- DB 스키마·테이블명

## 2단계: 실제 코드베이스 대조

각 요소에 대해:

- 파일·패키지·클래스: Grep·Glob으로 실제 존재 여부 확인
- 어노테이션: 실제 사용처의 타입(JPA? Reactive? WebFlux?)과 일치하는지 확인
- 피처 플래그 key: 같은 문서 내 key 스타일 일관성 + 기존 application.yml 관례와 일치 여부
- 성능 수치: 실측 근거가 있는지, 추측인지 판단. 실측 근거 없으면 "추측"으로 플래그
- 의존성: build.gradle·package.json에 실제 존재하는지
- DB 스키마: 마이그레이션 파일·엔티티에서 실제 컬럼·테이블 확인

## 3단계: 불일치 리포트

결과를 다음 표로 정리:

| 항목 | 문서의 가정 | 실제 상태 | 심각도 | 수정 제안 |
|---|---|---|---|---|
| (예) ConceptRepository 테스트 | @DataJpaTest | Neo4j Reactive | High | @DataNeo4jTest 또는 @SpringBootTest로 교체 |

심각도 기준:
- High: spec을 그대로 실행하면 실패가 확실한 항목
- Medium: 실행은 되지만 의미가 왜곡되는 항목 (예: 추측 수치를 기준선으로 굳히는 것)
- Low: 네이밍 불일치 등 사소한 정리 항목

## 4단계: 갱신 제안

3단계 결과를 바탕으로 마일스톤 문서 수정안을 제시. 단, 실제 수정은 내 승인 후에만 수행.

- High 항목은 반드시 수정 제안
- Medium 항목은 "이 수치는 Task X.X에서 실측으로 대체 필요"로 플래그 추가 제안
- Low 항목은 수정안만 제시하고 선택 맡김

## 5단계: Spec 분해 가능 여부 판단

마지막에 다음 중 하나로 결론:
- "Spec 분해 가능": High 0건, Medium 수정·플래그 완료
- "수정 후 재실행 필요": High 항목 존재. 문서 갱신 후 /survey-milestone 재실행

## 금지 사항

- 문서를 그대로 신뢰하고 바로 spec 분해로 넘어가지 말 것
- 추측 수치를 "기준선"으로 간주하지 말 것
- 실제 코드 확인 없이 "문서가 맞다"고 결론 내지 말 것
