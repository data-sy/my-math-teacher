# 그래프 데이터베이스를 RDB 의 재귀 SQL 로 대체하기

> MMT 서비스의 마일스톤 2 회고 — Neo4j 그래프 탐색을 MySQL 재귀 CTE 로 옮긴 마이그레이션 보고서

**기간:** 2026-04-24 ~ 2026-05-12 (3주, 1인 개발)
**범위:** 검증 단계까지 (점진 출시·폐기는 후속 마일스톤으로 분리)
**핵심 결과:** 그래프 탐색 응답 시간 **p95 14.034 ms → 0.556 ms (약 25배 개선)**, 운영 DB 1종 감축 가능 상태로 도달

---

## 한 줄 요약

수학 개념의 선후 관계를 저장하는 데이터를 두 종류의 데이터베이스(MySQL + Neo4j) 에 중복으로 두던 구조에서, **그래프 탐색을 관계형 DB 의 재귀 쿼리로 옮겨 단일 DB 운영이 가능한 상태로 만들었다.** 성능은 오히려 빨라졌고, 결과의 정확성·시각화 안전성·알고리즘 의미 보존을 모두 자동화 테스트로 검증했다.

---

## 배경: 왜 마이그레이션이 필요했나

### 도메인 — 수학 개념의 선후 관계

이 서비스는 학생의 수학 취약점을 진단해 맞춤 학습 경로를 제시한다. 핵심 데이터는 **개념과 개념 사이의 "선수(prerequisite) 관계"** 다. 예를 들어 "100까지의 수 쓰고 읽기" 라는 개념을 이해하려면 먼저 "9까지의 수", "수의 순서", "1 큰 수와 1 작은 수" 같은 선수 개념이 갖춰져야 한다. 이 관계는 자연스럽게 **방향 그래프** 형태가 된다.

진단 결과 화면에서는 학생이 약한 개념을 노드로 표시하고, 그 위로 거슬러 올라가는 선수 개념 트리를 시각화해서 보여준다. 즉 화면 한 번 그리려면 "특정 노드에서 시작해 깊이 N 까지의 모든 선수 노드를 가져오는 그래프 탐색" 이 필요하다.

### 초기 구조의 문제

v1 (2023~2024) 에서는 이 그래프 탐색을 위해 **그래프 전용 데이터베이스 Neo4j** 를 도입했다. 그래프 탐색에 강점이 있는 DB 라는 점이 매력적이었다. 그러나 시간이 지나면서 다음 문제가 누적됐다:

1. **데이터 중복.** 개념의 메타데이터(이름, 설명, 단원 정보)는 MySQL 에, 같은 개념의 그래프 노드는 Neo4j 에 양쪽으로 존재. 데이터 정합성 책임이 분산되고, CSV 로 초기 로드한 뒤 한쪽만 갱신되면 어긋날 위험.
2. **인프라 복잡도.** 운영 환경(AWS) 에 MySQL + Neo4j 두 인스턴스를 띄워야 했고, 로컬 개발자도 Docker Compose 에서 컨테이너를 추가로 띄워야 했다. 백업·모니터링·접근 권한도 두 벌.
3. **비용.** AWS 그래프 호스팅 비용이 매월 누적.
4. **기술 부채 — 리액티브 강제.** Neo4j 의 공식 Spring 통합이 리액티브 API 만 제공해서, 다른 곳은 동기 코드인데 그래프 호출만 `Mono`/`Flux` 였다. 결국 호출부에서 `.block()` 으로 동기로 변환하는 안티패턴이 박혀 있었다.
5. **사용 빈도가 낮다.** 그래프 탐색은 진단 결과 화면에서만 호출됨. 24시간 단위로 캐싱하면 충분.

이 모든 문제는 같은 결론을 가리켰다: **Neo4j 만큼의 그래프 특화는 이 서비스에 필요 없다.** 일반 RDB 가 제공하는 재귀 쿼리(Recursive CTE) 로 충분히 같은 작업이 가능하다.

### 가설

- MySQL 8 의 `WITH RECURSIVE` 와 적절한 인덱스로 깊이 3~5 의 그래프 탐색을 충분히 빠르게 처리할 수 있다.
- 결과 캐싱(Redis, 24시간 TTL) 으로 진단 페이지 부하를 추가 흡수.
- 단일 DB 운영으로 데이터 정합성·인프라·비용·코드 단순화를 한 번에 회수.

## 접근: 점진적·롤백 가능 마이그레이션

운영 중인 핵심 기능을 통째로 바꾸는 위험을 피하기 위해 **세 단계로 쪼갰다.**

### 1단계: 안전망 구축 (선행 마일스톤 M1)

본 작업에 들어가기 전에 별도 마일스톤에서 다음을 먼저 갖췄다:
- **성능 기준선 측정.** 옛 Neo4j 경로의 응답 시간을 깊이별로 정밀 측정해 기록.
- **결과 스냅샷.** 옛 Neo4j 의 그래프 탐색 결과(5개 시작 노드 × 깊이 2/3/5 = 15 케이스) 를 JSON 으로 저장. 새 경로의 결과와 비교할 정답지.
- **피처 플래그 인프라.** 새 경로와 옛 경로를 한 번에 켜고 끌 수 있는 설정 키.
- **테스트 인프라.** Testcontainers 로 격리된 MySQL · Neo4j 통합 테스트 환경.

> 마이그레이션의 가장 큰 실패 원인은 "비교할 정답지가 없는 것" 과 "롤백 경로가 없는 것" 이다. 본 작업은 두 가지를 먼저 갖춘 위에서 시작했다.

### 2단계: 데이터 레이어 — 재귀 SQL 도입

새 그래프 탐색을 MySQL 의 `WITH RECURSIVE` 로 구현했다. 기본 구조는 다음과 같다:

```sql
WITH RECURSIVE prerequisite_path AS (
    SELECT concept_id, 0 AS depth
    FROM concepts WHERE concept_id = ?      -- 시작 노드

    UNION ALL

    SELECT next.concept_id, prev.depth + 1
    FROM prerequisite_path prev
    JOIN knowledge_space edge ON prev.concept_id = edge.from_concept_id
    JOIN concepts next ON edge.to_concept_id = next.concept_id
    WHERE prev.depth < ?                    -- 최대 깊이 제한
)
SELECT concept_id, MIN(depth) AS depth
FROM prerequisite_path
GROUP BY concept_id;
```

`MIN(depth)` 는 같은 노드에 여러 경로로 도달했을 때 **가장 짧은 거리**를 채택한다는 의미. 이게 그래프 의미상의 "최단 거리" 와 같다.

성능을 위해 `knowledge_space` 테이블에 `(from_concept_id, to_concept_id)` 복합 인덱스를 추가했다. 재귀 단계마다 일어나는 `JOIN ... ON prev.concept_id = edge.from_concept_id` 와 `SELECT edge.to_concept_id` 가 이 인덱스로 **covering** 처리되어 행 페치 비용이 사라진다.

### 3단계: 서비스 레이어 — 분기와 캐싱

서비스 계층의 그래프 탐색 메서드 5개에 피처 플래그 분기를 적용했다. 같은 입력에 대해:

- 플래그 OFF → 기존 Neo4j 경로
- 플래그 ON → 새 재귀 SQL 경로 (+ Redis 캐싱, 24시간 TTL)

이때 한 가지 미묘한 함정이 있었다. 옛 코드 일부는 그래프 탐색 결과 ID 리스트를 받아 별도의 BFS 유틸로 거리 맵을 계산했다 — **그런데 이 BFS 유틸은 ID 리스트가 "Neo4j 가 반환하는 경로 순서대로 정렬되어 있다"** 는 전제로 동작했다. 새 SQL 경로는 결과를 정렬·중복 제거(`GROUP BY`) 해서 반환하므로 이 전제가 깨진다. 같은 입력 처럼 보여도 결과가 조용히 잘못될 위험이 있었다.

이 문제를 사전에 발견하고 해결책으로 **거리 맵을 SQL 단계에서 직접 산출하는 헬퍼**를 새로 만들었다. 재귀 CTE 가 이미 각 노드의 `MIN(depth)` 를 계산하니, 이를 그대로 `Map<id, distance>` 로 변환하면 된다. BFS 유틸을 거치지 않으니 경로 순서 가정과 분리된다. 양쪽 분기 모두 이 헬퍼를 통과하도록 코드를 정리.

## 검증: 무엇을 어떻게 확인했나

마이그레이션의 옳음을 다음 네 가지 차원에서 자동화 테스트로 확인했다.

### (1) 결과의 정확성 — 옛 경로와 같은 답을 내는가

15 케이스의 회귀 테스트를 작성했다. 각 케이스는 "시작 노드 X 에서 깊이 D 까지의 선수 개념 집합" 을 옛 경로(M1 에서 저장해둔 정답지) 와 새 경로(현재 SQL) 둘 다에서 산출해 비교한다.

**비교 방식의 미묘함:** 옛 Neo4j 경로의 결과는 "여러 경로로 같은 노드에 도달한 횟수까지 포함된 multiset" 이고, 새 SQL 경로의 결과는 "중복 제거된 unique set" 이다. 같은 그래프를 다른 형태로 표현하는 것뿐이라 의미적으로는 같지만, 형태가 다르니 `containsExactlyInAnyOrder` 같은 단순 비교는 항상 실패한다. 따라서 다음 두 단계로 비교한다:

1. 양쪽을 **unique set 으로 정규화**한 뒤 일치 여부 비교
2. 정규화된 결과의 SHA-256 해시를 재계산해 다시 한 번 비교 — 정규화 절차 자체의 무결성 가드

15 케이스 모두 통과.

### (2) 성능 — 옛 경로 대비 어떤가

같은 시작 노드(서비스 도메인에서 가장 연결도가 높은 "99까지의 수 쓰고 읽기" 개념) 에 대해 양쪽 경로의 응답 시간을 비교했다. 측정 방식은 동일: warm-up 3회 + 측정 100회, p95/p99 산출.

| 깊이 | 옛 경로 (Neo4j) p95 | 새 경로 (MySQL CTE) p95 | 배수 |
|-----:|--------------------:|------------------------:|-----:|
| 2 | 8.289 ms | 0.433 ms | 약 19× 빠름 |
| 3 | 14.034 ms | 0.556 ms | 약 25× 빠름 |
| 5 | 12.898 ms | 0.471 ms | 약 27× 빠름 |

> 깊이 5 가 깊이 3 보다 빠른 이유: 시작 노드의 서브그래프가 깊이 3 에서 이미 거의 다 탐색되어 깊이 5 에서 추가 작업이 적기 때문 + JIT 워밍업 효과. 양쪽 경로 모두 같은 노드를 사용했으므로 두 경로의 비교 의미는 그대로 유지된다.

당초의 가설은 "허용치 안에만 들어오면 OK" 였다 — 깊이 3 은 30 ms 이하, 깊이 5 는 100 ms 이하. 실제로는 두 자릿수 배율로 빨라졌다. 이유는 두 가지로 추정:

1. **사용 패턴 적합도.** Neo4j 는 임의 깊이의 그래프 탐색·중심성 분석 등 복잡한 워크로드에 최적화돼 있다. 본 서비스의 워크로드는 "유한 깊이의 단방향 트리 탐색" 으로 훨씬 단순해서, 일반 RDB 의 인덱스 기반 JOIN 으로도 충분하고 오히려 빠르다.
2. **데이터 규모.** 1,631 노드 / 3,446 엣지 — Neo4j 가 진가를 발휘하는 수십만~수백만 규모와는 거리가 멀다.

### (3) 시각화 안전성 — 프론트 화면이 깨지지 않는가

프론트는 Cytoscape.js 라이브러리로 그래프를 렌더링한다. 백엔드가 반환하는 응답 형태(JSON 의 필드 셋) 가 달라지면 시각화가 깨진다. 다음을 검증:

- 노드 응답에 `conceptId`, `conceptName`, 단원 정보 등 시각화에 필요한 필수 필드가 모두 포함됨
- 엣지 응답이 source/target ID 를 빠짐없이 가짐
- 응답 노드 집합이 (1) 의 회귀 스냅샷과 일치 — 시각화될 노드 개수·구성이 옛 경로와 같음

### (4) 알고리즘 의미 보존 — 거리 맵이 옛 의미와 같은가

학습 확률 계산에서 "시작 노드에서 각 선수 노드까지의 최단 거리" 가 입력값으로 쓰인다. 새 SQL 경로의 `MIN(depth)` 값이 이 "최단 거리" 의미를 보존하는지 확인:

- 시작 노드의 거리 = 0
- 모든 거리값이 0 이상 최대 깊이 이하
- 결과 노드 집합이 회귀 스냅샷과 일치
- 시작 노드를 제외한 모든 노드가 적어도 한 번은 거리 맵에 등장

### 부하 측면 — k6 시나리오

검증 테스트와 별개로 HTTP 엔드포인트 부하 시나리오를 작성해 두었다 (`shared/performance-tests/`). 운영자가 실제 환경에서 실행할 때 두 모드로 분기해 비교할 수 있다:

- **warm 모드:** 첫 호출이 캐시 미스, 이후 캐시 히트. 진단 페이지의 실제 사용 패턴.
- **cold 모드:** 매 요청 전 캐시 무효화 → 모든 호출이 캐시 미스. 캐시가 무력화된 최악의 경우.

본 검증 단계에서는 시나리오 작성과 문법 검증까지만 완료, 실제 부하 실행은 운영자의 후속 작업.

## 의사결정 기록

마이그레이션 도중 굳혀진 비-자명한 결정들. 각 결정은 별도 ADR(Architecture Decision Record) 로 보관.

| ADR | 주제 | 핵심 결정 |
|---|---|---|
| 0001 | 테스트 커버리지 선행 | 마이그레이션 전에 결과 스냅샷·성능 기준선·피처 플래그 구조를 먼저 구축. 비교할 정답지 없는 마이그레이션 금지 |
| 0003 | 캐싱 패턴 | Spring Cache 추상화 도입 대신 Redis 유틸 직접 호출. 사용 메서드 수가 적고(5개) 키 네임스페이스 통제가 더 명확. 추상화 도입은 사용처가 늘었을 때로 미룸 |
| 0004 | 모니터링 인프라 분리 | Prometheus + Grafana 같은 production 모니터링은 본 마이그레이션 범위 밖. 검증 단계는 `SimpleMeterRegistry` + 로그 기반 측정으로 충분. dashboard·alerting 은 후속 마일스톤 |
| 0005 | 객체 반환 SQL 의 매핑 | 그래프 탐색 결과를 도메인 객체로 채울 때 `concepts` 와 `chapters` 테이블을 JOIN. 일부 필드(섹션 매핑) 는 프론트 미사용이라 매핑 생략 — 사용처 기반 정합화 |

### 작업 분할 정책

- **마이그레이션 자체는 데이터 레이어 → 서비스 레이어 → 검증** 3단계로 분리해 각각 별도 PR. 한 PR 이 너무 커지면 리뷰가 형식적이 되고 회귀 원인 추적이 어려워진다.
- **검증 단계 내부**도 회귀(정확성)·성능·시각화·거리 맵 의미를 각각 다른 테스트 클래스로 분리. 한 클래스가 여러 책임을 가지면 실패 원인 파악 비용이 늘어난다.
- **출시·관찰·폐기**(피처 플래그 ON → 1 개월 관찰 → Neo4j 인프라 제거) 는 시간 척도가 검증 작업의 수십 배라 같은 마일스톤에 묶이면 모호한 closed scope 가 된다. 이를 별도 마일스톤 M3 로 분리하고, 본 마일스톤 M2 는 "검증된 마이그레이션" 으로 닫음.

## 회수한 것

- **응답 시간 약 25배 개선** (깊이 3 기준 14 ms → 0.6 ms). 본래 기대치는 "허용치 안" 정도였으나 두 자릿수 배율 회수.
- **운영 DB 1종 감축 가능 상태**. 인프라·백업·모니터링·접근 권한이 한 벌로 줄어듦.
- **데이터 중복 원인 제거.** 개념 데이터를 한 DB 에만 두면 정합성 책임이 분산되지 않는다.
- **리액티브 안티패턴(`.block()`) 제거 경로**. 그래프 호출만 동기로 변환하던 코드가 자연 해소되며, 코드베이스가 일관된 동기 모델로 회귀.
- **알고리즘 함정 사전 발견.** "ID 리스트가 경로 순서라는 옛 BFS 유틸의 숨겨진 전제" 를 마이그레이션 도중 발견. 만약 발견 못 했다면 거리 맵이 조용히 잘못된 값을 내는 silent regression 으로 이어졌을 가능성.

본 보고서 시점에서 다음은 **별도 마일스톤(M3)** 로 분리됐다.

- 피처 플래그 ON 으로 운영 환경 점진 전환
- 1 개월 무사고 관찰 (캐시 히트율 >90%, 에러율 0%)

- Neo4j 빌드 의존성·도메인 어노테이션·logging 설정 제거
- Docker Compose 에서 Neo4j 컨테이너 제거 + ADR 동반 (인프라 변경 정책)

## 부록: 메타 회고

이번 작업에서 의식적으로 따른 규칙 몇 가지.

### "Analyze-Before-Change"

스키마 변경·레이어 간 리팩토링 같은 비가역 작업 직전에 항상 다음을 먼저 정리하고 사용자 승인을 받았다 — 영향 받는 호출 지점, 영향 받는 테스트 목록, 롤백 시나리오. 어떻게 보면 단순한 체크리스트이지만, 이걸 강제하지 않으면 "일단 손대고 보자" 가 되어 변경 범위가 부풀거나 회귀 원인을 잃기 쉽다.

### "트레이드오프 결정은 사용자에게"

기술적 선택 중 정답이 하나 있는 사실 확인은 직접 진행했지만, 양쪽 다 합리적인 선택지가 있는 트레이드오프(예: 점진 전환의 옵션 A/B/C, 캐시 측정 방식 선택, 회귀 테스트의 위치) 는 옵션·권장·근거를 정리해 사용자에게 결정 받았다. 이렇게 분리하면 진행 속도를 잃지 않으면서도 중요한 결정의 권한이 분명해진다.

### "spec 본문은 작업 중에도 살아 있는 문서"

검증 작업 중 spec 본문의 옛 가정(스냅샷 날짜, 코드 라인 번호, 양쪽 분기 비교의 실현 가능성) 이 어긋난 부분을 발견할 때마다 작업 흐름을 멈추고 spec 본문을 먼저 정정했다. spec 이 코드와 동기화돼 있어야 spec 을 보고 작업 가능 — 그렇지 않으면 spec 은 곧 죽은 문서가 된다.
