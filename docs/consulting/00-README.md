# 리디자인 & React 이주 컨설팅 프롬프트 묶음

MMT(My Math Teacher) 실서비스 배포 전, 2년 묵은 디자인을 전반적으로 손보고
Vue→React 이주 타당성·비용을 평가하기 위한 **컨설팅 프롬프트 모음**이다.

여기 든 프롬프트는 **이 세션에서 실행하지 않는다.** 각각을 새 세션(빈 컨텍스트)에서
독립적으로 붙여넣어 돌리는 용도다. 두 프롬프트는 서로 의존하지 않으므로 병렬로 돌려도 된다.

## 왜 둘로 쪼갰나 (페르소나 분리 근거)

두 일은 평가 축과 전문성이 완전히 다르다. 한 세션에 섞으면 디자인 취향 판단과
엔지니어링 견적이 서로의 가정을 오염시킨다.

| | 01 · 디자인/UX 컨설팅 | 02 · Vue→React 이주 스코핑 |
|---|---|---|
| 페르소나 | 시니어 프로덕트 디자이너 (UX 리서치 + 비주얼 + 디자인 시스템) | 시니어 프론트엔드 아키텍트 (프레임워크 이주 리드) |
| 판단 축 | "지금 무엇이 안 좋고, 어떤 경험으로 바꿔야 하나" | "React로 옮길 가치가 있나 + 옮기면 리소스가 얼마인가" |
| 출력 | 디자인 비평 + 리디자인 방향 + 디자인 시스템 권장안 | 이주 여부 판단 + AI 자율주행 기준 견적 + 단계 계획 |
| 코드 변경 | 없음 (프레임워크 무관, 비주얼/UX/DS만) | 없음 (분석·견적·계획만, 실제 이주 코드는 별도) |

## 확정된 전제 (이 묶음을 만들 때 정한 것)

- **디자인은 프레임워크 무관**으로 진행 — Vue/React 어느 쪽 구현에도 적용 가능한
  비주얼·UX·디자인 시스템 산출물만 낸다. (→ 01)
- **이주 프롬프트는 "여부 판단 + 견적"** — React로 옮길지 trade-off부터 따지고,
  옮긴다면 AI 에이전트 방식(사용자 지시 → Claude 코딩) 기준 리소스를 견적한다. (→ 02)
- **디자인 시스템은 열어둔다** — 페르소나가 Prime 유지 / Tailwind+shadcn / MUI 등을
  비교·권장하고 최종 결정은 사용자가 한다.

## 실행 순서 권장

1. **01과 02를 각각 다른 세션에서 병렬로** 돌린다. (의존 없음)
2. 02의 "이주 여부 판단"이 나오면, 그 결론에 따라 디자인 결과(01)를 어느 스택에서
   구현할지 결정한다. — 디자인 자체는 프레임워크 무관이라 02 결과를 기다릴 필요 없음.
3. 두 산출물이 모이면, 이 세션(또는 새 세션)으로 돌아와 통합 실행 계획을 세운다.
   그때 비로소 spec/브랜치/`/analyze-before-change`로 넘어간다.

## 산출물을 어디에 저장할지

각 프롬프트는 결과 보고서를 `docs/consulting/out/` 아래에 저장하도록 지시한다.
(없으면 그 세션에서 생성) 실제 코드 변경은 이 단계에서 하지 않는다.
