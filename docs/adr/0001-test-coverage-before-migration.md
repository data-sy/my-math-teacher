# ADR 0001: 마이그레이션 전 테스트 커버리지 선행 구축

## Status

Accepted

## Context

프로젝트는 그래프 탐색 기능을 Neo4j Reactive 기반으로 운영 중이며, 인프라 단순화와 운영 비용 감소를 위해 MySQL 재귀 CTE로 이전하는 마이그레이션(Milestone 2)이 예정되어 있다.

현재 코드베이스의 상태는 다음과 같다:

- 통합 테스트 커버리지가 부족하여, 마이그레이션으로 인한 회귀를 자동으로 감지할 수 있는 기반이 없음
- 성능 측면의 기준선이 문서화되어 있지 않아, 이전 후 성능 저하가 발생해도 이를 정량적으로 판정할 근거가 없음
- 피처 플래그나 조건 분기 구조가 없어, 마이그레이션 배포 후 문제 발생 시 즉시 롤백할 수단이 없음
- 그래프 탐색 결과의 동치성(Neo4j 결과 = MySQL CTE 결과)을 검증할 스냅샷이 없음

이러한 상태에서 마이그레이션을 직접 착수하면, 기능 회귀나 성능 저하를 **운영 중에 사용자로부터 발견**하게 될 위험이 크고, 발견한 뒤에도 롤백 판단·실행 모두 수동으로 이루어져 피해 확대가 예상된다.

## Decision

Milestone 2(마이그레이션) 착수 전에 **Milestone 1을 선행**하여 다음 네 가지 기반을 구축한다:

1. **Testcontainers 기반 통합 테스트 환경**
   - 마이그레이션 전후의 기능 동등성을 자동 검증할 수 있는 테스트 인프라
   - 그래프 쿼리 3종(선수 개념, 깊이 N 탐색, BFS 경로) 각각에 대한 테스트

2. **성능 기준선 실측 및 기록**
   - 주요 쿼리별 평균·p95·p99 응답 시간을 실측 (warm-up + 반복 측정)
   - 결과를 문서화하여 M2에서 회귀 판정의 절대 기준으로 사용

3. **그래프 탐색 결과 스냅샷**
   - Neo4j의 현재 결과를 JSON으로 보존
   - M2 완료 후 MySQL CTE 결과와 바이트 레벨 비교로 동치성 검증

4. **피처 플래그 및 조건 분기 구조**
   - `mmt.migration.use-mysql-cte-for-graph` 등 플래그 도입
   - 서비스 레이어에 조건 분기 구조 준비하여 즉시 롤백 가능한 배포 단위 확보

이 네 가지는 Milestone 1의 3개 Spec(Testcontainers, 성능 기준선, 피처 플래그)으로 분할되어 실행된다.

## Consequences

### Positive

- 마이그레이션 후 회귀를 자동 감지 가능 (통합 테스트·결과 스냅샷 비교)
- 성능 저하 발생 시 실측 기준선과 정량적으로 비교하여 판정 가능
- 피처 플래그로 즉시 롤백 가능. 문제 발견 즉시 플래그 토글만으로 기존 동작 복원
- 이 마일스톤에서 구축된 테스트 인프라·플래그 구조는 이후 JdbcTemplate → JPA 전환 Epic에서도 재사용됨

### Negative

- Milestone 2 착수 시점이 약 3일 지연됨
- 테스트 코드·벤치마크 코드가 추가되어 빌드 시간과 CI 실행 시간이 증가
- 테스트 데이터 생성·컨테이너 기동 비용으로 인해 개발 루프가 약간 느려질 수 있음

### Neutral

- 이 마일스톤은 이후 모든 마이그레이션성 작업의 전제가 되므로, 구축된 자산은 M2 종료 후에도 계속 유지됨
- 피처 플래그 네임스페이스(`mmt.<영역>.*`) 컨벤션이 확정되어 향후 플래그 추가 시 일관성 확보됨

## Alternatives Considered

### 1. 마이그레이션과 테스트를 병행 작성

M2를 착수하면서 필요한 테스트를 그때그때 작성하는 방식.

**기각 이유:**
- 테스트의 목적은 "마이그레이션 전 상태"를 고정하는 것인데, 병행 작성하면 이미 변경된 코드에 대한 테스트가 되어 기준선 역할을 못 함
- 개발자 주의가 마이그레이션과 테스트로 분산되어 양쪽 모두 품질 저하 위험

### 2. 기존 수동 검증만으로 진행

별도 테스트 없이 개발자가 수동으로 시나리오를 검증하고 배포.

**기각 이유:**
- 수동 검증은 커버리지가 작성자 주관에 의존
- 회귀가 발생해도 어느 쿼리가 원인인지 특정하기 어려움
- 성능 회귀는 수동으로 측정 자체가 비현실적

### 3. Dry-run 데이터베이스 비교만 수행

MySQL CTE 쿼리를 병행 실행하여 결과를 Neo4j와 비교하는 방식.

**기각 이유:**
- 결과 동치성은 검증할 수 있지만 성능·안정성은 커버 못 함
- 피처 플래그 없이는 롤백 불가
- 테스트 자동화가 없어 회귀 감지가 실시간이 아닌 사후적
